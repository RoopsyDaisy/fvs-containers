## 🔴 1 — The "Output .xlsx for current run" button crashes

> **RESOLVED 2026-06-02 (Option B).** Patched via
> `patches/fvsOL-rsqlite-temp-tables.patch` (now 12 sites) + the broadened guard
> in `apply_fvsol_patch.sh`, and committed to the `ForestVegetationSimulator-Interface`
> branch `claude/amazing-curie-QxCIu`. The chosen fix is the temp-table form
> (Option B), whose `temporary=TRUE` mechanism is already proven on the pinned env
> (it's what the other 11 sites + `smoke_test.R` guard 2 use) — so no pinned-env
> repro was needed for unit correctness. **Still to do:** the GUI click-through
> (build a webgui image from the fix, run a projection, click the button) as the
> end-to-end confirmation — see "How to verify (b)" below. The analysis is kept
> below as the record.

**Symptom (the bug, now fixed).** In the WebGUI, after running a projection, clicking
**"Output .xlsx for current run"** throws

```
Error: Named parameters not used in query: name
```

— the *exact* RSQLite ≥3 error the `fvsOL-rsqlite-temp-tables.patch` eliminates at
the other 11 sites. The recent work declared that error class handled "to all 11
sites"; this is a **12th instance in a user-reachable path** that the patch
intentionally scoped out (see `patches/README.md` → "Intentionally NOT converted").

**Where.** `vendor/fvs-interface` (pinned `57b9aa3`) → `fvsOL/R/server.R:4829`,
inside the `output$dlFVSOutxlsx` download handler. It is wired to a real button:
`fvsOL/R/ui.R:962` `downloadButton("dlFVSOutxlsx", "Output .xlsx for current run")`.

```r
# server.R ~4826
tmp        = paste0("tmp", gsub("-","",runuuid), Sys.getpid(), "genoutput")
dbExecute(dbGlb$dbOcon, paste0("attach database ':memory:' as ", tmp))
casesToGet = paste0(tmp, ".casesToGet")
dbWriteTable(dbGlb$dbOcon, name = DBI::SQL(casesToGet), value = cases, overwirte = TRUE)   # <-- line 4829
...
dbExecute(dbGlb$dbOcon, paste0("detach database ", tmp, ";"))                              # line 4848
```

Two defects on that one line:
1. **`name = DBI::SQL(casesToGet)`** — passing *any* `DBI::SQL(...)` as the `name`
   arg is what triggers the error on RSQLite ≥3 (your pinned 3.53.1). The 11
   patched sites used the literal `DBI::SQL("temp.X")` form; this one is the
   *dynamic* form (`casesToGet` is the attached-schema-qualified name
   `tmp….casesToGet`), but the failing mechanism is identical.
2. **`overwirte = TRUE`** — a pre-existing typo; it's swallowed into `...`, so
   `overwrite` silently stays `FALSE`.

The attached `:memory:` db (`tmp`) is used **only** for `casesToGet` and is
detached at line 4848, so the fix is well-contained.

### Why no test caught it

`scripts/smoke_test.R` guard 2 exercises the temp-table *pattern in isolation*
(it writes its own `Grps` table), never the actual fvsOL handlers; the integration
test runs the FVS **CLI**, not the GUI. So nothing drives `dlFVSOutxlsx`. (Same
"false green" the `patches/README.md` notes for the 5 non-`server.R` sites — it
applies here too.) See finding #2.

### The fix

Pick **A** (smallest, preserves upstream structure) and fall back to **B** if A
doesn't write to an attached schema on the pinned RSQLite. Either way, **fix the
`overwirte` typo**.

**Option A — `DBI::Id()` (preferred).** Keep the attach/detach; just stop wrapping
the name in `SQL()`:

```r
dbWriteTable(dbGlb$dbOcon,
             name  = DBI::Id(schema = tmp, table = "casesToGet"),
             value = cases, overwrite = TRUE)
```

`Id(schema=…, table=…)` is the modern DBI idiom for a qualified name; RSQLite emits
`CREATE TABLE "tmp"."casesToGet"`, which is valid against the attached alias. The
downstream `select … from <casesToGet>` (still `tmp….casesToGet`) is unchanged.

**Option B — temp table (fallback).** If A doesn't land in the attached schema,
drop the attach entirely and use the proven pattern from the 11 patched sites
(known-good on the pinned env — smoke guard 2):

```r
# remove the `attach database ':memory:' as tmp` (line 4827) and the matching
# `detach` (line 4848); then:
dbExecute(dbGlb$dbOcon, "drop table if exists temp.casesToGet")
dbWriteTable(dbGlb$dbOcon, "casesToGet", cases, temporary = TRUE, overwrite = TRUE)
casesToGet = "temp.casesToGet"     # downstream `from temp.casesToGet` resolves
```

### Also harden the completeness guard

`scripts/apply_fvsol_patch.sh`'s guard greps only for the **literal**
`dbWriteTable(…, (name=)?DBI::SQL("temp.…` form, so it neither catches this
dynamic site nor would catch a future regression of it. After fixing 4829,
broaden the guard to flag **any** `DBI::SQL()` passed as a `dbWriteTable` name —
e.g.

```sh
grep -rnE 'dbWriteTable\([^,]+,[[:space:]]*(name[[:space:]]*=[[:space:]]*)?DBI::SQL\(' "$SUB/fvsOL/R"
```

Post-fix this matches nothing (the 11 sites use plain strings; `server.R:7712` is
a plain string; 4829 is now `Id()`/temp). Confirm zero matches before shipping, so
the guard locks the whole error class — not just the `"temp."` literal — out.

### Where the change lands

The fix lives in vendored upstream, so mirror the existing patch workflow:
1. Extend **`patches/fvsOL-rsqlite-temp-tables.patch`** with the 4829 hunk (the
   patch is a diff against pinned `57b9aa3`, where 4829 still has the old form).
2. Update **`patches/README.md`** (move 4829 out of "Intentionally NOT converted"
   into the converted set; note the `overwirte` typo fix) and the guard above.
3. Optionally mirror the same edit into the `ForestVegetationSimulator-Interface`
   repo's `upstream-pr/fvsol-rsqlite-temp-tables` branch for upstreaming.

### How to verify (lab PC)

**(a) Pinned-env live repro** — fastest; decides A vs B without the GUI. In R with
the pinned RSQLite (3.53.1):

```r
library(DBI); library(RSQLite)
con <- dbConnect(SQLite(), ":memory:")
dbExecute(con, "attach database ':memory:' as tmp1")
cases <- data.frame(CaseID = 1:3)

# reproduce the bug (expect: Error: Named parameters not used in query: name)
try(dbWriteTable(con, name = DBI::SQL("tmp1.casesToGet"), value = cases))

# Option A — does Id() write into the attached schema?
dbWriteTable(con, name = DBI::Id(schema = "tmp1", table = "casesToGet"),
             value = cases, overwrite = TRUE)
stopifnot(dbGetQuery(con, "select count(*) n from tmp1.casesToGet")$n == 3)  # A works
```

If the `stopifnot` passes, ship Option A. If `Id()` errors or writes to the wrong
schema, use Option B.

**(b) GUI end-to-end** — the real confirmation. Either build/run the WebGUI image
(`docker run --rm -p 3838:3838 -v "$PWD/proj:/work" fvs-webgui:ie`) or, in the
devcontainer, `bash scripts/run_webgui.sh`; then at `http://localhost:3838`: load a
project, run an FVS projection, and click **"Output .xlsx for current run"**.
Success = a non-empty `.xlsx` downloads and the R console shows no "Named
parameters…" error. **The repo owner can drive this if the agent can't run a
browser session** — ask them to do the click-through against a webgui image built
from the fix branch.

## 🟠 2 — No end-to-end WebGUI test (OPEN, improvement)

The publish gate (`scripts/build_images.sh` → smoke + integration) never exercises
a real fvsOL handler, which is why #1 (and the 5 non-`server.R` sites before it)
could sit unnoticed. `scripts/smoke_test.R:8` points at "the separate shinytest2 +
Chromium suite" as the faithful end-to-end check — **that suite does not exist in
this repo** (only the comment). Options, cheapest first:
- A headless **boot** check: launch `fvsOL()` against the bundled training data,
  assert the Shiny app object builds and `getVolumes2()`/startup don't error
  (catches the class of "session aborts on open" bugs without a browser).
- The referenced **shinytest2 + Chromium** click-through (run a projection, hit
  each download incl. the `.xlsx`) — this is what would have caught #1.

Until one exists, either build it or drop the dangling "separate suite" reference
in `smoke_test.R` so it doesn't imply coverage that isn't there.

## ✅ 3, 4, 5 — fixed this pass

- **#3 (toolkit):** `cluster/fvs_run_one.sh` now honors a `BIND` env var
  (`--bind`), and `cluster/fvs_array.sbatch` defaults `BIND=/mnt/beegfs` and
  exports it — matching `fvs_rfvs_array.sbatch`. Previously the CLI array staged
  the shared input DB as an *absolute symlink* and ran `apptainer exec` with no
  `--bind`; on Hellgate beegfs is a default bind so it worked, but on a portable
  Apptainer site where the shared FS isn't auto-mounted the symlink would dangle
  and FVS couldn't open the DB. shellcheck-clean.
- **#4 (toolkit):** `tests/unit/test_data_paths.R` now saves/unsets `FVS_DATA_DIR`
  for its duration (restored on exit). The examples tell users to
  `export FVS_DATA_DIR=…`; running the suite in that shell used to fail
  `data_paths/reads-present` spuriously (CI passed only because it never set it).
- **#5 (doc drift, both repos):** this README's "Testing & contributing" no longer
  claims the unit tests run in-image (they moved to the toolkit); the
  `.github/agents/*.agent.md` playbooks (implementer/data-explorer/qa) no longer
  reference the pruned Python stack (`src/`, pytest, black/ruff); `PROJECT_STATUS.md`
  Key-files table + CI line updated; `.pre-commit-config.yaml` header corrected

## ⚪ 6 — `server.R:7712` dot-named temp table (latent, pre-existing)

`dbWriteTable(dbGlb$dbOcon, "temp.FVS_ClimAttrs", climd, overwrite=TRUE)` writes a
table *named* `temp.FVS_ClimAttrs` in the **main** schema, not a temp-schema table,
so a later `… from temp.FVS_ClimAttrs` (schema-qualified) would miss it. Already
flagged in `patches/README.md` as out of scope for the RSQLite-error patch; tracked
here so it isn't lost. Lower priority than #1 (climate-attributes path; different
failure mode — silent, not a crash). If you touch it, it's a separate small patch.
